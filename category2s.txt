### **System Goal**  
Design a **hierarchical category system** similar to Digikala, with the ability to define **parent categories** and **subcategories**, along with **product management**, such that:

- Each product is linked to **at least one category** (either parent or child).
- **Smart filtering** based on category hierarchy is supported.

---

### **1. Entities Definition**

| Entity | Fields |
|--------|--------|
| **Category** | `id`, `name`, `slug`, `parent_id` (nullable), `level` (integer: 0 for parent, 1 for child, etc.), `is_active` |
| **Product** | `id`, `name`, `slug`, `price`, `status`, `category_id` (direct category assignment) |

---

### **2. Category Business Rules**

- Each category can be either a **parent** (`parent_id = null`) or a **child** (`parent_id != null`).
- There is **no strict limit** on depth (but 3 levels are sufficient in practice).
- A parent category can have **multiple children**.
- A child category can have **only one parent**.

---

### **3. Product–Category Assignment Rules (Critical Point)**

- A product **can be assigned directly to a parent category** (without being linked to any child).
- A product **can be assigned to a child category** (which itself belongs to a parent).
- **Important**: If a product is assigned to a child category, it **must also be considered a member of its parent category** for filtering purposes (logical inheritance).

---

### **4. Filtering Logic**

When a user filters by a **parent category**:

- Show **all products** that are **directly assigned** to that parent category.
- **Also show** all products that are assigned to **any child category** of that parent (recursively).

> **In other words:** filtering by a parent should **automatically include all descendants**.

---

### **5. Database Implementation Suggestions (for Performance)**

To implement this filtering efficiently (without heavy recursive queries), one of these two approaches is recommended:

- **Path Enumeration:**  
  Add a `path` field (e.g., `"1/3/5/"`) to each category and use `LIKE` queries to fetch all descendants.

- **Closure Table:**  
  Create a separate table that stores all ancestor–descendant relationships for fast filtering.
