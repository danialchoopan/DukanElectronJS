2. Data Architecture

2.1 Database

SQLite via better-sqlite3 in WAL mode. All tables defined in connection.ts with CREATE TABLE IF NOT EXISTS.

Key rule: All columns must be in CREATE TABLE. Only use ALTER TABLE inside migrateSchema() for auto-migration of old databases.

2.2 Repository Pattern

Each table has a dedicated repository file. Functions use getDatabase() directly:

// Example: products.ts
export function getAllProducts(): Product[] {
  const db = getDatabase()
  return db.prepare('SELECT * FROM products WHERE isActive = 1').all().map(formatProduct)
}

2.3 IPC Pattern

All UI↔DB communication goes through ipcMain.handle() in handlers.ts. UI calls via window.api (preload):

// main process (handlers.ts)
ipcMain.handle('products:getAll', () => products.getAllProducts())

// renderer (component)
const r = await window.api.products.getAll()
if (r.success && r.data) setProducts(r.data)

2.4 Double-Entry Accounting

Every financial transaction automatically posts to the general journal:

3. Auto-Migration

The migrateSchema() function runs on every app startup:

// 1. PRAGMA table_info() — read existing columns
// 2. Compare against expected columns list
// 3. ALTER TABLE ADD COLUMN for any missing ones
// 4. Idempotent — no-op after first migration
When adding a new column: Add it to both CREATE TABLE in connection.ts AND expectedColumns in migrateSchema().

7. Adding a New Feature

  1. Add new columns to CREATE TABLE in connection.ts
  2. Add columns to migrateSchema() for auto-migration
  3. Update types in types/index.ts
  4. Update the relevant repository (formatProduct, create, update)
  5. Add IPC handler in handlers.ts
  6. Add preload method in preload/index.ts
  7. Build or update the UI component
  8. Verify: npx tsc --noEmit + node scripts/verify.js
